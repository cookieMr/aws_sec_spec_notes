**Overview**: AWS Key Management Service (KMS) is a managed service for creating and controlling encryption keys used to protect data. KMS is integrated with over 70 AWS services and provides centralized key management, audit trails, and fine-grained access control. It is the foundation of data protection in AWS and one of the most heavily tested services on the exam.

**Domain weight**: KMS is the most important service in the Data Protection domain (~18% of SCS-C03) and appears across nearly every domain — encryption, key policies, and integration scenarios are tested extensively.

## 1. Key Types

### 1.1. Customer Master Keys (CMKs)

| CMK Type             | Origin               | Can Be Viewed? | Can Be Managed? | Rotation                                   | Cost                     |
| -------------------- | -------------------- | -------------- | --------------- | ------------------------------------------ | ------------------------ |
| **AWS managed**      | AWS KMS              | No             | No (limited)    | Automatic (every 3 years)                  | Free                     |
| **Customer managed** | AWS KMS or imported  | Yes            | Yes             | Optional (annual or imported key rotation) | $1/month + usage         |
| **Custom key store** | AWS CloudHSM cluster | Yes            | Yes             | Manual (key material in CloudHSM)          | $1/month + CloudHSM cost |

- **AWS managed keys**: Created automatically by AWS services (e.g., S3 creates a key for SSE-S3, but KMS has `aws/s3` keys when using SSE-KMS). You cannot view the key material, modify the key policy, or rotate the key manually.
- **Customer managed keys**: You create, manage, rotate, and control access to these keys. Full control via key policies + IAM.
- **Custom key store (CloudHSM)**: Keys are stored in a CloudHSM cluster under your control. Required for mandates that keys be stored in FIPS 140-2 Level 3 validated HSM.

**Exam scenario**: A compliance requirement mandates that encryption keys be stored in a dedicated HSM under the customer's direct control → use **custom key store** backed by **AWS CloudHSM**.

### 1.2. Symmetric vs Asymmetric CMKs

| Feature             | Symmetric                               | Asymmetric                                             |
| ------------------- | --------------------------------------- | ------------------------------------------------------ |
| **Encrypt/Decrypt** | Yes                                     | Yes (RSA)                                              |
| **Sign/Verify**     | No                                      | Yes (RSA, ECC)                                         |
| **Key material**    | 256-bit AES key                         | RSA or ECC key pair                                    |
| **API**             | `Encrypt`, `Decrypt`, `GenerateDataKey` | `Sign`, `Verify`, `Encrypt`, `Decrypt`                 |
| **Use case**        | Server-side encryption in AWS services  | Digital signatures, client-side encryption outside AWS |

- Symmetric keys are the default and most common — used by integrated AWS services
- Asymmetric keys are used when encryption happens **outside AWS** (no KMS API access) or when you need digital signatures

## 2. Envelope Encryption

### 2.1. Why Envelope Encryption

- KMS has a **4 KB limit** per encrypt/decrypt operation — you cannot encrypt data larger than 4 KB directly
- Solution: Use **envelope encryption** — encrypt data with a **data key**, and encrypt the data key with a KMS CMK
- The data key is a symmetric key (256-bit) generated by KMS

### 2.2. How It Works

1. You call `GenerateDataKey` — KMS returns a **plaintext data key** (for encryption) and an **encrypted data key**
2. You encrypt your data (any size) with the plaintext data key
3. You discard the plaintext data key (only keep the encrypted data key)
4. To decrypt: send the encrypted data key to KMS — KMS decrypts it with the CMK and returns the plaintext data key
5. You decrypt your data with the plaintext data key

### 2.3. Key APIs

| API                               | What It Does                                               |
| --------------------------------- | ---------------------------------------------------------- |
| `GenerateDataKey`                 | Returns a plaintext data key + encrypted data key          |
| `GenerateDataKeyWithoutPlaintext` | Returns only the encrypted data key (for storage-only use) |
| `GenerateDataKeyPair`             | Returns plaintext + encrypted asymmetric key pair          |
| `Decrypt`                         | Decrypts encrypted data keys (up to 4 KB)                  |

## 3. Key Policies

### 3.1. Policy Types

| Policy Type    | Scope                                           | Max Size                       |
| -------------- | ----------------------------------------------- | ------------------------------ |
| **Key policy** | Attached directly to the CMK (resource-based)   | 32 KB                          |
| **IAM policy** | Attached to a user/role                         | 6 KB (inline), 10 KB (managed) |
| **Grants**     | Alternative to policy — allows temporary access | No policy size limit           |

### 3.2. Key Policy vs IAM Policy (How Access Works)

KMS access requires **both** the key policy AND the IAM policy to allow the action (unless the key policy explicitly delegates to IAM).

- **Default key policy**: Gives the AWS account full control (account root user can manage the key) and delegates permission management to IAM policies. With this setting, any IAM policy that grants KMS actions will work.
- **Custom key policy**: You define exactly who can access the key. If you remove the account-wide delegation, IAM policies alone will NOT grant access — the principal must also be listed in the key policy.

**Exam scenario**: An IAM user has `kms:Decrypt` in their IAM policy but still gets AccessDenied on a KMS key → the **key policy** may not grant the user permission. KMS requires both the key policy AND the IAM policy to allow the action.

### 3.3. Key Policy Example

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "Enable IAM policies",
      "Effect": "Allow",
      "Principal": {
        "AWS": "arn:aws:iam::123456789012:root"
      },
      "Action": "kms:*",
      "Resource": "*"
    }
  ]
}
```

This default policy delegates access control to IAM. Without this, you must list every principal in the key policy directly.

## 4. Grants

### 4.1. Purpose

- A mechanism to **grant temporary access** to a KMS key without modifying the key policy
- Used by **integrated AWS services** that need to use your KMS key on your behalf

### 4.2. How Grants Work

- The key administrator creates a grant using `CreateGrant`
- The grant specifies: principal, operations allowed, optional encryption context constraints
- Grants have no policy size limit (unlike key policies which have 32 KB limit)
- Grants can be revoked by the grantee or expire

### 4.3. Grant vs Key Policy

| Aspect          | Grant                                 | Key Policy                    |
| --------------- | ------------------------------------- | ----------------------------- |
| **Persistence** | Temporary (can be revoked)            | Permanent (until modified)    |
| **Size limit**  | None                                  | 32 KB                         |
| **Use case**    | AWS services (e.g., EBS, RDS, Lambda) | Administrative access control |
| **Management**  | Can be created/revoked by key admins  | Must be modified by key admin |

**Exam scenario**: An AWS service (e.g., S3 or Lambda) needs to use your KMS key to encrypt/decrypt data → the service creates a **grant** on the key with the necessary permissions.

## 5. Encryption Context

### 5.1. Purpose

- A set of key-value pairs that is cryptographically bound to the ciphertext
- Acts as **additional authenticated data (AAD)** — if the context does not match, decryption fails
- Provides an additional layer of security and audit trail

### 5.2. How It Works

```bash
aws kms encrypt \
  --key-id alias/my-key \
  --plaintext fileb://secret-data \
  --encryption-context "department=finance,environment=prod"
```

- To decrypt: you must provide the **same** encryption context
- Encryption context is logged in CloudTrail (visible for audit)
- Can be used in key policies with the `kms:EncryptionContextKeys` condition

**Exam scenario**: A security policy requires that specific KMS keys can only be used to encrypt/decrypt data for specific applications → use **encryption context** with `kms:EncryptionContextKeys` and `kms:EncryptionContext:*` conditions in the key policy.

## 6. Key Rotation

### 6.1. Automatic Rotation (Customer Managed CMKs)

- Can enable **automatic annual rotation** — KMS rotates the key material once per year
- Old key material is retained for decryption of previously encrypted data
- The key ID, ARN, and alias remain the same — no application changes needed
- The backing key changes — new encryption uses the new backing key
- **Not available** for imported key material or asymmetric keys

### 6.2. Rotation for Imported Key Material

- If you import key material (instead of letting KMS generate it), automatic rotation is **not supported**
- You must manually rotate by creating a new key, importing new material, and updating the alias

### 6.3. AWS Managed Keys

- Rotated automatically every **3 years** (cannot be configured)
- No user control over rotation schedule

**Exam scenario**: A compliance policy requires annual key rotation for all encryption keys → enable **automatic key rotation** on customer managed CMKs.

## 7. Key States and Lifecycle

| State                  | Description                                            | Can Be Used? |
| ---------------------- | ------------------------------------------------------ | ------------ |
| **Enabled**            | Key is active                                          | Yes          |
| **Disabled**           | Key is inactive                                        | No           |
| **Scheduled deletion** | Key will be deleted after a waiting period (7-30 days) | No           |
| **Pending import**     | Key material is waiting to be uploaded                 | No           |
| **Unavailable**        | Key in custom key store is unavailable                 | No           |

- Deleting a key is irreversible — all data encrypted with that key becomes permanently undecryptable
- Deletion has a **waiting period** (7-30 days) that can be cancelled

**Exam scenario**: Data encrypted with a KMS key is suddenly not decryptable → check the **key state** — the key may have been disabled or scheduled for deletion.

## 8. Key Aliases

- A friendly name for a KMS key: `alias/my-key`
- Aliases can be **updated to point to different keys** — enabling key rotation without changing application code
- Example: Application uses `alias/prod-db-key` — when you rotate the key, point the alias to the new key
- Different keys in different regions can share the same alias name

## 9. Multi-Region Keys

- KMS keys that can be replicated to other AWS Regions
- The key in each region has the same key ID, key material, and key policy
- Can be used independently in each region — no cross-region dependency for decryption
- Use case: Global applications that need encryption keys in multiple regions

## 10. Cross-Account Access

KMS keys can be used across accounts in two ways:

| Method         | How It Works                                      | Use Case                                               |
| -------------- | ------------------------------------------------- | ------------------------------------------------------ |
| **Key policy** | Add the external account to the key policy        | Another account needs direct access to the key         |
| **Grants**     | Delegate access to an IAM role in another account | An AWS service in another account needs to use the key |

**Exam scenario**: Account A has a KMS key. Account B needs to encrypt/decrypt data using it → Account A adds Account B's root user to the **key policy**, and Account B creates an IAM policy granting the specific KMS actions.

## 11. KMS Limits

| Resource                          | Limit                                               |
| --------------------------------- | --------------------------------------------------- |
| Encrypt/Decrypt request rate      | 5,500-50,000 per second (per key, varies by region) |
| Data key generation rate          | Same as Encrypt/Decrypt                             |
| Max data size per Encrypt/Decrypt | 4 KB                                                |
| Key policy max size               | 32 KB                                               |
| Grants per key                    | 10,000                                              |
| Aliases per region per account    | 100                                                 |
| Customer managed keys per region  | 10,000                                              |
| Automatic rotation schedule       | Annual (once per year)                              |

## 12. KMS Integration with AWS Services

| Service             | How KMS Is Used                                                            |
| ------------------- | -------------------------------------------------------------------------- |
| **S3**              | SSE-KMS for server-side encryption of objects                              |
| **EBS**             | Encrypt EBS volumes and snapshots                                          |
| **RDS**             | Encrypt RDS databases and read replicas                                    |
| **Lambda**          | Encrypt environment variables; Lambda execution role needs KMS permissions |
| **SQS**             | Encrypt queue content                                                      |
| **SNS**             | Encrypt topic content                                                      |
| **Secrets Manager** | Encrypt secrets                                                            |
| **DynamoDB**        | Encrypt tables at rest                                                     |
| **CloudTrail**      | Encrypt log files (SSE-KMS on S3)                                          |
| **CloudWatch Logs** | Encrypt log group data                                                     |

## 13. KMS vs CloudHSM

| Feature         | AWS KMS                          | AWS CloudHSM                                   |
| --------------- | -------------------------------- | ---------------------------------------------- |
| **Management**  | Fully managed (AWS handles HSM)  | Customer-managed HSM cluster                   |
| **Control**     | AWS controls the HSM             | Customer controls the HSM (FIPS 140-2 Level 3) |
| **Scalability** | Automatic                        | Manual (provision HSMs)                        |
| **Integration** | 70+ AWS services (built-in)      | No direct AWS service integration              |
| **API**         | KMS API (Encrypt, Decrypt, etc.) | PKCS#11, JCE, Microsoft CSP                    |
| **Use case**    | Most AWS encryption needs        | Compliance requiring dedicated HSM             |

**Exam tip**: KMS for almost everything. CloudHSM only when you need direct control of the HSM, FIPS 140-2 Level 3, or PKCS#11/JCE interface for custom applications.

## 14. Security Best Practices

- **Use automatic key rotation** for customer managed CMKs
- **Use encryption context** to bind key usage to specific data
- **Use key policies + IAM policies** together (default delegation pattern)
- **Use grants** for temporary access (AWS services)
- **Use aliases** for transparent key rotation
- **Restrict key deletion** with IAM policies
- **Monitor KMS API calls** in CloudTrail — especially `DisableKey`, `ScheduleKeyDeletion`
- **Use `kms:ViaService` condition** to restrict key usage to specific AWS services
- **Use `kms:EncryptionContextKeys` condition** to enforce encryption context requirements
- **Separate keys per environment** (dev, test, prod) and per application

## 15. Exam Tips

1. **KMS encrypts up to 4 KB directly** — for larger data, use envelope encryption with GenerateDataKey.

2. **Key policy + IAM policy**: Both must allow the action for KMS access (unless the key policy delegates to IAM).

3. **Key rotation**: Automatic annual rotation for customer managed CMKs. Not supported for imported key material.

4. **Encryption context** is AAD — cryptographically bound to ciphertext. Must match to decrypt. Logged in CloudTrail.

5. **Grants** are used by AWS services to use your KMS key on your behalf.

6. **Multi-region keys** have the same ID and material across regions — useful for global applications.

7. **Cross-account KMS**: Use key policy to grant cross-account access.

8. **CloudHSM vs KMS**: KMS for most scenarios. CloudHSM only when you need direct HSM control or specific compliance.

9. **Symmetric keys** are the default and used by AWS services. **Asymmetric keys** for external encryption or signing.

10. **Key states**: Enabled (active), Disabled (inactive), Scheduled deletion (pending deletion, 7-30 day wait).

11. **Deleting a key is permanent** — all data encrypted with it becomes undecryptable.

12. **Aliases** enable transparent key rotation — point the alias to the new key, no application changes needed.

13. **kms:ViaService** restricts key usage to a specific AWS service (e.g., `kms:ViaService`: `s3.amazonaws.com`).

14. **kms:CallerAccount** condition restricts key usage to a specific AWS account.

15. **CloudTrail** logs all KMS API calls — monitor for `DisableKey`, `ScheduleKeyDeletion`, and failed `Decrypt` calls.

16. **Data key caching** (via KMS Encryption SDK) reduces calls to KMS by caching data keys in memory.

17. **KMS key ARN format**: `arn:aws:kms:<region>:<account>:key/<key-id>`. Always use the full key ARN in IAM policies, not key ID.

18. **Symmetric keys have ID format**: `mrk-` (multi-region) or `1234abcd-12ab-34cd-56ef-1234567890ab`.
